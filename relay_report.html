Executive Summary

Relay is a lightweight, self-hosted Continuous Integration engine built in Python using FastAPI. It ingests GitHub webhook events, validates them cryptographically, and routes them through a priority-aware job queue that dispatches work to a pool of language-specific workers. Each worker executes a project's custom pipeline definition (a minimal YAML DSL called .relay.yml), streams real-time log output to a SQLite-backed store, and surfaces build status through a browser dashboard with 4-second auto-refresh polling.

Key Technical Achievements

  • HMAC-SHA256 webhook verification with constant-time comparison and idempotent delivery deduplication — duplicate GitHub redeliveries are detected via a UNIQUE constraint on delivery_id and returned as status: "duplicate" without creating a new job.
  • Multi-dimensional priority scoring — push to default branch scores ~290+ (critical), hotfix/release branches ~275+ (high), pull-request synchronise events ~135 (normal), ensuring release-critical builds are never queue-starved by feature-branch churn.
  • Thread-safe scheduler using threading.RLock and threading.Event that dispatches jobs concurrently across four workers, wakes within 350 ms of a new arrival, and recovers interrupted jobs on restart.
  • Four-worker pool with per-language affinity (Python, Node, Java/Go, and a universal fallback) load-balanced via composite score with random jitter to prevent hot-worker clustering.
  • Live traffic simulator generating synthetic webhook payloads on configurable intervals (2–6 s default) for stress-testing without a real SCM integration.

Architecture Summary

Relay is a modular monolith — a single FastAPI process containing six functional modules (Ingress, Queue, Orchestration, Execution, Persistence, Delivery) that communicate via in-process Python calls. Persistence is handled by a single SQLite database with three tables: repositories, jobs, and job_logs.

Top 3 Lessons Learned

  • Thread safety at queue boundaries is non-trivial. The first version used dict.pop() without a lock — two worker threads occasionally claimed the same job simultaneously. Wrapping reservation in RLock eliminated the issue.
  • A priority queue without deduplication creates silent duplicates. Redelivered GitHub webhooks caused the same commit to be built twice. A single UNIQUE constraint on delivery_id fixed it.
  • Simple polling beats complex push for a first iteration. An earlier design explored Server-Sent Events; a 4-second fetch interval is simpler, requires no persistent connection management, and is visually indistinguishable from streaming.

System Architecture

2.1 Architecture Overview

Relay is a modular FastAPI monolith. Rather than deploying separate network services, each functional boundary is a Python module within a single process. This keeps deployment trivially simple (uvicorn main:app) while preserving the logical separation of concerns that would make decomposition feasible in a future iteration.

2.2 Module Decomposition

Webhook Ingress

  • Tier: Ingress
  • Responsibility: Receive raw HTTP POST from GitHub, validate HMAC signature, normalise payload, hand off to job creation.
  • Inbound: POST /webhooks/github with headers x-github-event, x-github-delivery, x-hub-signature-256 and a JSON body.
  • Returns: HTTP 202 with {jobId, triggerType, priorityLabel, queueStatus}.

Event Decider

  • Tier: Ingress. Pure function — no I/O.
  • Accepts: push (all), pull_request with action in {opened, synchronize, reopened}.
  • All other events (ping, release, star, …) return should_create=False with a reason string.

Priority Engine

  • Tier: Orchestration. Pure function — no I/O.
  • Push scoring: 200 + branch_score (10–90) + clamped commit count (max 20) + distinct-author bonus (max 9).
  • Pull-request scoring: 125–145 based on action — always below push scores.
  • Fallback score: 50 for unknown trigger types.

Scheduler

  • Tier: Orchestration.
  • Background daemon thread draining the priority queue by matching jobs to available workers.
  • Wake-event mechanism: schedule_job() sets a threading.Event, waking the loop within 350 ms instead of waiting for the 1 s timeout.
  • Startup recovery: marks all assigned/processing jobs as failed before starting the dispatch loop.

Pipeline Design (.relay.yml)

3.1 Pipeline File Structure

Relay defines its own minimal YAML-like DSL stored in .relay.yml at the root of each monitored repository. This was a deliberate trade-off: full Groovy/Jenkins DSL parsing would have consumed the majority of the project timeline, while a simple sequential step runner delivers the core CI value proposition with a fraction of the complexity.

name: python-api-pipeline
language: python

steps:
  - name: Install dependencies
    command: pip install -r requirements.txt

  - name: Run linter
    command: flake8 app/

  - name: Run tests
    command: pytest tests/ -v

Supported Directives

  • name (top-level): Human-readable pipeline identifier, written to job logs.
  • language: Optional hint; if omitted the system uses the repository's registered language.
  • steps: Ordered list of build steps. Each step requires name and command.

Features intentionally NOT implemented

  • Parallel stages — de-scoped in favour of multi-worker parallelism at the queue level.
  • When conditionals — branching logic handled at job-creation layer (Event Decider).
  • Post handlers — job status + log stream serve as the equivalent signal.

3.2 Parsing & Pipeline Loading

The parser (app/services/pipeline.py) is hand-rolled — no external library. A single linear pass over the file:

  • Lines beginning with # or blank are skipped.
  • Lines before steps: are parsed as top-level key-value pairs (colon delimiter).
  • After steps:, lines beginning with - start a new step; subsequent indented lines are step properties.
  • Quoted values ("..." or '...') have quotes stripped at parse time.
Validation rules: at least one step required · every step needs both name and command · malformed lines (no colon) raise ValueError caught by the executor, which transitions the job to failed.

3.3 Execution Model

Pipeline execution is strictly sequential — steps run one at a time in declaration order.

  • Executor calls update_job_status(job_id, 'processing') and logs the worker assignment.
  • A simulated startup delay (0.15–0.75 s × speedMultiplier) models real-world container warm-up.
  • For each step: subprocess.Popen launches the command in the workspace directory with shell=True.
  • Two daemon threads capture stdout (level=info) and stderr (level=error) line-by-line and write each to job_logs.
  • process.wait() blocks until the child exits; both log threads are joined to avoid lost output.
  • Non-zero exit: log the failure, call update_job_status(job_id, 'failed'), skip remaining steps.

Component Deep Dives

4.1 Webhook Listener & Ingress

HMAC-SHA256 Verification

The _verify_signature helper computes the expected signature:

expected = 'sha256=' + hmac.new(SECRET.encode('utf8'), raw_body, hashlib.sha256).hexdigest()

Uses hmac.compare_digest for constant-time comparison, preventing timing-based secret extraction. A length check fires first to reject obviously invalid signatures cheaply.

Idempotency

Before creating a new job, the handler queries get_job_by_delivery_id(delivery_id). If a job already exists, it returns HTTP 200 with status: "duplicate" and the existing job's ID. The database enforces uniqueness via UNIQUE on jobs.delivery_id — a hard guarantee even under concurrent redeliveries.

Payload Normalisation

build_github_job_context(event, payload) normalises push and pull_request payloads into a canonical context dict: repository, trigger_type, ref, commit_sha, pull_request_number, action, base_ref, head_ref. Downstream components never touch the raw payload.

4.2 Job Scheduler & Priority Queue

Priority Scoring

score = 200 + branch_score + min(commit_count, 20) + min(distinct_authors * 3, 9)

Queue Data Structure

The queue is the jobs table itself. Dequeue is:

SELECT ... FROM jobs WHERE status = 'in_queue'
ORDER BY priority_score DESC, created_at ASC LIMIT 1

A composite index on (status, priority_score DESC, created_at ASC) keeps dequeue O(log n). Jobs are not deleted — they transition status, leaving a durable audit trail.

Startup Recovery

_recover_queue_on_startup scans for assigned/processing jobs and marks them failed before the dispatch loop starts — preventing jobs from remaining in a pseudo-running state indefinitely after a server restart.

Reservation Algorithm

  • Pass 1: workers whose supportedLanguages contains the job's language and currentJobId is None.
  • Pass 2 (fallback): if Pass 1 is empty, expand to workers accepting "generic".
  • Selection: lowest composite score = jobs_completed + exact_match_penalty + random jitter (0–0.35).

Random jitter prevents hot-worker clustering when multiple workers share the same jobs_completed count (common at startup).

4.4 Multi-Language Build Runners

Runner selection is implicit: worker pool language affinity handles routing. The executor is language-agnostic — it runs whatever shell commands are in .relay.yml. The "runner interface" is the pipeline file itself.

4.5 Observability & Dashboard

Single HTML page (~255 lines Jinja2) hydrated by a 537-line vanilla JavaScript file. No frontend framework or bundler.

Data Fetching (every 4 s)

  • GET /health — service status
  • GET /repositories — registered repos with verification status
  • GET /jobs — full job list with logs and priority metadata
  • GET /queue — live queue snapshot (queued, active, worker states)
  • GET /workers — worker pool state

XSS prevention: all user-originated strings pass through escapeHtml() before innerHTML assignment.

5.2 Retry & Recovery Mechanisms

  • Job retry: not implemented — CI systems should be deterministic. Automatic retries on build failures would mask genuine test failures.
  • Worker unavailability: job stays in_queue with priority intact; scheduler retries dispatch within 1 s or immediately when a worker finishes.
  • DB integrity: all writes use explicit BEGIN/COMMIT transactions with rollback on exception, serialised by threading.RLock.

5.3 Failure Injection Observations

  • Worker crash: exception caught by _run_assigned_job wrapper → job marked failed, worker released, next queued job dispatched within 350 ms.
  • Duplicate redelivery: redelivering the same x-github-delivery via curl returned HTTP 200 status:"duplicate". UNIQUE constraint provided the hard guarantee.
  • Missing pipeline file: executor logged "Pipeline file not found at …" and transitioned to failed within milliseconds; worker immediately available for the next job.
  • Invalid HMAC: tampered body consistently returned HTTP 401. No job created.

Scaling Strategies & Future Work

6.1 Current Scaling Limits

  • Worker concurrency: hard-coded at 4. Adding a fifth requires modifying WORKER_BLUEPRINTS and restarting.
  • Queue throughput: SQLite write-lock contention becomes a bottleneck above ~100 webhook events/second.
  • Stateful components: WorkerPool and Scheduler are in-memory — cannot be replicated across processes.
  • Log storage: no retention policy; verbose builds can generate thousands of job_logs rows per job.
  • Filesystem: workspaces are local paths; horizontal scaling would require a shared filesystem or per-job provisioning.

6.2 Containerisation & Orchestration

  • Containerised runners (proposed): replace Popen(shell=True, …) with docker run --rm -v workspace:/workspace image sh -c command. Pre-pull images once per worker start.
  • Kubernetes model (proposed): pipeline stages map to K8s Jobs (batch/v1). Scheduler becomes a controller watching a RelayJob Custom Resource. HPA on queue-depth metric triggers scale-up.

Reflections & Learnings

7.1 Technical Growth

Hardest Component

The scheduler. The challenge was not the dispatch logic but the interaction between threading primitives: ensuring the wake event, the RLock, and the stop event composed correctly under all race conditions — particularly "shutdown while a job is dispatching." Several iterations were needed before the lock scope was tight enough to avoid deadlocks while still wide enough to protect invariants.

Design Decision We Would Reverse

The static WORKER_BLUEPRINTS dict. It got us to a working system quickly, but worker capacity cannot change without a code change and restart. We would replace it with a database-backed worker registry allowing workers to self-register via a heartbeat API.

Biggest Surprise

How much complexity lives in the delivery-ID idempotency requirement. GitHub webhook redelivery is not an edge case — it happens whenever GitHub's delivery confirmation times out. Without the UNIQUE constraint, load testing showed that aggressive redelivery could create 3–5× more jobs than actual commits. The fix was one SQL constraint; recognising the need required actually triggering the failure.

From Jenkins User to Builder

Having used Jenkins as a black box, building Relay forced us to reason about every abstraction it hides: how does a push become a queued job? How does a job get matched to an executor? How does subprocess stdout end up in a dashboard log? Each answer is a non-trivial engineering decision that Jenkins makes on your behalf — building it from scratch made those decisions, and their consequences, visible.

7.2 DevOps Philosophy

Building Relay reframed the Dev vs Ops tension as a data-flow problem. Developers own the .relay.yml (what to run); the system owns the when, where, and how. This boundary — a versioned file in the repository rather than a GUI configuration — means the pipeline is reviewed in pull requests and testable locally. That insight is what "pipeline as code" means in practice.

The most concrete lesson: reliability is not a feature you add at the end. The startup recovery, the idempotency constraint, and the transaction rollbacks were each added in response to a specific failure observed during development. A production CI system cannot afford that discovery process in production.

7.3 Team Dynamics

The system was divided along module boundaries: one team member owned the ingress layer (webhooks, event decider, priority engine), the other owned the execution layer (scheduler, worker pool, executor, pipeline parser). The dashboard and persistence layer were built jointly.

Integration challenge: the interface between scheduler and executor was initially informal. When the executor started returning structured error dicts instead of raising exceptions, the scheduler had to be updated. Defining a formal return contract ({ok: bool, …}) before writing implementations would have prevented this mismatch.
